# FioVoucher

FioVoucher is a YAML-driven voucher plugin for Minecraft servers. The current `1.8` source registers the Bukkit plugin as `FioVouchers` and provides voucher redemption, voucher item distribution, trial voucher handling, confirmation menus, player history menus, configurable storage, Bedrock handling, and a public Java API.

This page is based on the current project files under `src/main/java` and `src/main/resources`, including `plugin.yml`, `config.yml`, the command handler, storage classes, menu managers, bundled voucher YAML files, and `FioVoucherAPI`.

## Verified Feature Set

### Player Features

The player-facing flow is intentionally small:

- `/voucher redeem <code>`
- `/voucher trials`
- `/voucher pause <code>`
- voucher item redemption through the item listener
- confirmation menu before the claim is executed
- active trial and player history menu views

### Voucher Features

The voucher system supports:

- file-based voucher definitions from `plugins/FioVoucher/vouchers/*.yml`
- one-time, limited-use, or unlimited-use vouchers through `settings.redeem.max-claim`
- fixed reward actions
- weighted random reward groups through `commands.random-claimed`
- item reward actions through the `[ITEM]` action prefix
- trial vouchers
- frozen vouchers
- disabled vouchers
- per-voucher world restrictions
- per-voucher region restrictions
- permission whitelist and blacklist checks
- duplicate voucher item protection
- voucher item protection rules for inventories, containers, item frames, armor slots, and selected environment damage cases
- item metadata examples for custom model data, armor trims, leather colors, potion metadata, banner or shield patterns, bundle contents, glow, unbreakable items, flags, names, and lore

### Runtime Features

The runtime layer includes:

- YAML storage with MySQL support and YAML fallback
- player claim history and paused trial state
- language files in `lang/*.yml`
- configurable command aliases
- Bedrock redeem control and optional confirmation bypass
- hibernation and idle wake handling
- confirmation, voucher history, and player history GUI files
- player history and voucher history menus
- update checker notifications for permitted admins
- optional PlaceholderAPI support

## Command Surface

Main command:

```text
/voucher
```

Aliases from `plugin.yml`:

```text
/vouchers
/fiovoucher
/fvoucher
```

Additional aliases can be configured through `settings.command-aliases`.

Common usage:

```text
/voucher redeem <code>
/voucher trials
/voucher pause <code>
/voucher list
/voucher info <code>
/voucher admin give <player> <voucherid> [amount]
/voucher admin giveall <voucherid> [amount]
/voucher admin history <player>
/voucher admin freeze <voucherid>
/voucher admin unfreeze <voucherid>
```

Full permissions and command details are documented in [Commands & Permissions](/plugin/fiovoucher/commands-permissions).

## Redeem Flow

Players redeem a voucher code through `/voucher redeem <code>` or by using a voucher item. Before executing rewards, FioVoucher validates plugin readiness, Bedrock access, voucher existence, disabled state, frozen state, global blacklist rules, world restrictions, region restrictions, remaining uses, max-claim history, trial settings, and permission gates.

When validation succeeds, the plugin executes the configured voucher actions, records player history, decrements remaining uses when applicable, and saves voucher state.

## Voucher Files

File-based vouchers live in:

```text
plugins/FioVoucher/vouchers/*.yml
```

Each voucher file uses this main structure:

- `voucher.id`
- `voucher.display-name`
- `voucher.enabled`
- `voucher.item`
- `voucher.commands.random-claimed`
- `voucher.commands.claimed`
- `voucher.commands.failed`
- `voucher.commands.max`
- `voucher.settings.trial`
- `voucher.settings.redeem`

The bundled source contains `26` voucher YAML files. See [Voucher YAML List](/plugin/fiovoucher/vouchers/list) for the voucher ids, file names, materials, and usage notes.

## Storage

FioVoucher supports two storage modes:

- `YAML`
- `MYSQL`

The storage type is configured through `storage.type`. If MySQL initialization fails, the current storage implementation attempts to fall back to YAML storage.

## Developer API

The public `FioVoucherAPI` exposes:

- plugin readiness checks
- voucher lookup by code
- voucher list and voucher code access
- voucher item creation
- give voucher helpers for one player or all online players
- redeem validation through `canRedeem`
- redeem execution through `redeem`
- formatted player history access
